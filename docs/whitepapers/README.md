# HCW-UI white papers

The philosophy underneath the design system — *why* the interface is the way it is,
before any token or component. Read these to understand the system; read the
governance docs to enforce it.

| # | Paper | What it argues |
| --- | --- | --- |
| 01 | [**Depth Encodes Importance**](depth-encodes-importance.md) | The three material languages — **flat → soft → glass** — as a scale of demand. Why the z-axis carries meaning, why glass is scarce, and why most of the screen stays calm and flat. |
| 02 | [**The Calm Interface**](the-calm-interface.md) | The operating philosophy: design for the operator under pressure, not the explorer. Chrome serves the task, one spatial model, disclose the next step, work without a mouse. |
| 03 | [**Orchestration Lives in the Rail**](orchestration-lives-in-the-rail.md) | The core AI move: **content in the stage, orchestration in the rail, command at the bottom bar**. Why the chat window is the wrong shape for supervising an AI, and how HCW decomposes it. |
| 04 | [**AI-Orchestration UX**](../ai-orchestration-ux-framework.md) | Mission-first, not conversation-first. The human supervises intent; the AI manages execution. The framework the rail model implements. ([ESTI application](../esti/HCW-AI-ORCHESTRATION-UX.md)) |
| 05 | [**From the Report to the Model**](from-report-to-model.md) | The spatial turn: from **flat reports to data modelled on the thing itself** (anatomy on a body, data on the building). The **DOS→GUI** leap for information — and why calm chrome is what lets the model speak. |
